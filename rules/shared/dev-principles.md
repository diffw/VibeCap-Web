# 开发原则

1. **最大化单元测试覆盖率** — 为所有公共接口、边界条件和关键路径编写单元测试，追求尽可能高的代码覆盖率。
2. **精简代码，遵循最佳实践** — 代码应简洁清晰，遵循语言和框架的最佳实践，避免过度设计。任何时候要增加代码，都需同时看哪些已有的代码可精简。
3. **修复 Bug 必须定位根因** — 修复 Bug 前，必须先定位真正的 Root Cause，严禁未理解成因就盲目修改。但此条仅适用于 Bug 修复；新功能开发的范围由需求定义，不需要在项目中广泛排查"相关逻辑"。无论哪种任务类型，检索过程都必须遵循 Scope Card 的检索预算（见 `rules/shared/scope-guard.md`）。
4. **可测试性优先** — DI + 纯函数；代码变更必须同步调整测试并提高覆盖率。
5. **简洁至上，持续重构** — KISS/YAGNI/DRY，SOLID，小函数，组合优于继承；Boy Scout Rule，共性上移，个性下推。
6. **快速失败，禁止降级兜底** — 错误必须直接抛出并暴露，严禁通过 fallback 或降级逻辑绕过问题；尽早暴露故障才能定位并修复真正的 Root Cause。

## 测试策略

### 测试金字塔比例

```
Unit Tests        ▓▓▓▓▓▓▓▓▓▓▓▓▓▓  70%   — 快速、隔离、覆盖所有逻辑分支
Integration Tests ▓▓▓▓▓▓          20%   — 验证模块间交互契约
E2E Tests         ▓▓▓              10%  — 仅覆盖 5-10 条关键用户旅程
```

- 下层测试优先：能用 Unit 覆盖的场景不上移到 Integration，能用 Integration 覆盖的不上移到 E2E
- E2E 不是万能兜底——它慢、脆、维护成本高，仅用于验证关键业务流程的端到端可用性

### 覆盖率门槛

| 指标 | 最低要求 | 核心模块要求 |
|------|---------|------------|
| 行覆盖率 | 80% | 90%+ |
| 分支覆盖率 | 70% | 85%+ |

- 覆盖率下降的变更不应合并
- 核心模块定义：涉及收入、数据完整性、安全性的模块

### 层级执行顺序（强制）

```
Unit Tests 全部通过
    ↓
Integration Tests 全部通过
    ↓
E2E Tests 全部通过
```

- 上层测试的前置条件：下层测试必须全部通过
- 如果 Unit Tests 有失败，禁止跳过直接运行 Integration 或 E2E

## 交互质量通用原则

1. **三层契约** — 交互改动必须同时验证：语义正确、交互正确、时序稳定。
2. **幂等要求** — 重复事件（hover/click/pointer）不得引发状态漂移、定位跳变或可见性闪烁。
3. **回归闭环** — 每次交互 Bug 修复至少补两类回归：同层单元测试（状态机/映射）与上层 E2E（真实路径 + 短时稳定性断言）。
