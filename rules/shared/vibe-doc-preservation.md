# .vibe-doc Preservation

当任务会创建、更新或整理宿主项目的 `.vibe-doc/` 文档时，必须遵守以下保全规则。

## 核心原则

1. **不删除原有文档**
- 不得因为生成新文档而删除现有 `.vibe-doc/*` 文件。
- 不得把旧文档静默移动到别处后当作“已处理”。

2. **不以空内容或模板覆盖非空文档**
- 不得用空文件、半成品、占位模板直接覆盖已有非空文档。
- 不得把“先 copy 模板再慢慢填”用于已存在的非空目标文件。

3. **preserve-first**
- 如果目标文件已存在且非空，先读取它，再基于现有内容更新。
- 默认策略是**增量更新**，不是重建。

4. **显式替换必须有用户授权**
- 只有用户明确要求“重写 / replace / reset / 重新开始”时，才允许替换现有文档。
- 即使用户明确要求替换，也必须先说明影响范围，并保留可回溯版本。

5. **优先稳定 canonical path**
- 如果 canonical 文件存在且可继续使用，优先在原文件上更新。
- 只有在用户明确要求并行方案、或文档语义已经完全不同的情况下，才创建新的变体文件。

## 执行规则

### 初始化

- 只有目标文件缺失时，才允许从模板初始化。
- 如果目标文件存在但为空文件，可视为未初始化。
- 如果目标文件存在且非空，进入“更新模式”，而不是“初始化模式”。

### 更新模式

- 保留已有标题、已确认结论、人工补充内容、历史说明。
- 只补全缺失章节、修正明显冲突、或按用户要求更新对应段落。
- 若某段信息不确定，保留原文并标注待确认，不要直接抹掉。

### 新变体文件

只有满足以下任一条件时才创建新文件：

- 用户明确要求保留原稿并生成新版本
- 原文件是另一个 feature / 子主题的文档，不适合作为当前任务输出
- 任务本身要求产出带日期或带 feature 标识的报告

新文件命名应避免覆盖已有文件，例如：

- `PRD-<feature>.md`
- `product-spec-<platform>-<feature>.md`
- `ACCEPTANCE_REPORT_<feature>_<date>.md`

### JSON / 结构化文档

像 `design-token.json` 这类结构化文件：

- 默认做字段级更新，不整体重置。
- 保留未知字段、人工补充字段、已有 schema 扩展。
- 只有在用户明确要求 reset 时，才允许整体重建。

## 禁止行为

- 把已有 `.vibe-doc/*.md` 先清空，再写新内容
- 不读旧文件就直接 `copy template -> overwrite`
- 因为“当前任务只关心新功能”就删除旧 feature 文档
- 用空占位结果替换已有完整文档

## 完成前检查

如果本次任务触达 `.vibe-doc/`，完成前必须确认：

- 旧文档仍然存在
- 没有非空文档被空内容替换
- 新增文档的命名不会与已有文档冲突
- 若更新 canonical 文档，原有人工内容没有被无说明删除
