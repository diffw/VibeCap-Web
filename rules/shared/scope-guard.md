# Scope Guard — 范围锁定与分级验证

任何代码变更任务启动前，必须先输出 Scope Card 锁定范围，再动手写代码。

> **硬门禁**：Scope Card 是第一条工作消息。未输出 Scope Card 之前，禁止调用任何文件读写、搜索、编辑、Bash 工具。唯一允许的前置动作是读取本规则文件本身。

---

## Step 0：输出 Scope Card（强制前置）

收到需求后、写第一行代码前，先输出：

```
📋 Scope Card
─────────────────────────────
目标：  [一句话，做什么]
类型：  新功能 / Bug修复 / 重构 / 配置变更
风险区域：Critical / Standard / Low
改动规模：Micro / Small / Medium+
执行模式：Hard Budget / Soft Budget / Phase Gate
改动文件：[列出预计修改的文件，Micro ≤2，Small ≤3]
测试文件：[列出对应的测试文件]
验证计划：[跑哪些测试，用什么命令]
完成标准：[怎样算做完]
执行控制：[Hard/Soft 写预算；Phase Gate 写本阶段 blocking checks]
─────────────────────────────
```

Scope Card 一旦输出，后续工作必须严格在其范围内进行。需要扩大范围时，必须先向用户说明原因并获得确认。

---

## 风险区域定义

### Critical — 无论改动大小，必须完整验证

触碰以下区域的改动，即使只改 1 行，也不能走 Micro 快车道的最小验证：

- 支付 / 订阅 / IAP 逻辑
- 认证 / 会话 / 权限控制
- 数据迁移 / 删除 / 持久化 schema 变更
- 安全相关（沙箱、entitlements、加密、隐私）
- 核心数据模型的结构性变更
- 应用生命周期关键路径（启动、崩溃恢复）

### Standard — 验证深度随改动规模缩放

- UI 交互逻辑（键盘、鼠标、手势处理）
- 视图布局与样式
- 状态管理与数据流
- 导航与窗口管理
- 业务逻辑（非 Critical 类）

### Low — 微改动可最小验证

- 常量 / 配置值调整
- 日志 / 调试代码
- 代码注释
- 仅测试文件的变更
- 文案 / 本地化 key 修改（无逻辑变更）

---

## 改动规模分级

| 级别 | 代码行数 | 文件数 | 跨模块 |
|------|---------|--------|--------|
| Micro | ≤10 行 | 1-2 | 否 |
| Small | ≤50 行 | ≤3 | 可能 |
| Medium+ | >50 行 | >3 | 是 |

---

## 执行模式

不是所有任务都应该被同一种预算模型约束。  
本规则将执行控制分成三种模式：

### Hard Budget

适用于：

- `Micro` + `Low / Standard`
- 明确边界的小修复、小功能、小型文档更正

特点：

- 文件读取 / 搜索 / 测试次数采用**硬上限**
- 触达上限后，必须停下并申请扩范围
- 目标是防止 agent 把小任务做成无边界排查

### Soft Budget

适用于：

- 大多数 `Small` 任务
- 低到中等复杂度的功能补全、重构、脚本修补

特点：

- 预算是**预警阈值**，不是立刻停机的硬闸门
- 触达阈值后，必须显式说明为什么超出预期
- 如果要跨模块、换假设、扩大修改面，则必须重新锁定范围

### Phase Gate

适用于：

- `Medium+` 任务
- Bug 根因分析（RCA）
- flaky / 环境相关 / 跨模块问题
- 脚本、生成器、CI、目录迁移、harness / infra / meta 优化

特点：

- 不再以“读了几个文件、搜了几轮”作为主控制器
- 主控制器改为：**阶段边界 + blocking checks + phase receipt**
- 只有当前阶段 gate 通过，才能推进下一阶段

---

## 验证路径矩阵

| 规模 ╲ 风险 | Critical | Standard | Low |
|------------|----------|----------|-----|
| **Micro** | 目标测试 + 所属模块全量 | 仅目标测试 | 编译通过即可 |
| **Small** | 目标测试 + 所属模块全量 | 目标测试 + 相关模块 | 仅目标测试 |
| **Medium+** | 全量测试 | 所属模块全量 + 集成测试 | 目标测试 + 所属模块 |

**目标测试** = 只跑与改动直接相关的测试用例，不跑全量。例如：
```bash
xcodebuild test ... -only-testing:TestTarget/TestClass/testSpecificCase
```

## 预算与检查点

| 执行模式 | 文件读取 | 搜索轮次 | 测试执行 | 控制方式 |
|------|---------|--------|---------|---------|
| Hard Budget | 硬上限 | 硬上限 | 硬上限 | 触达即停 |
| Soft Budget | 预警阈值 | 预警阈值 | 预警阈值 | 触达后解释并重锁范围 |
| Phase Gate | 不设主硬限 | 不设主硬限 | 按阶段需要 | 以 gate / receipt / 晋级规则控制 |

### 默认预算建议

| 场景 | 默认执行模式 | 建议控制 |
|------|-------------|---------|
| `Micro` + `Low / Standard` | `Hard Budget` | 读 2 / 搜 2 / 测 1 |
| `Small` + 非 RCA | `Soft Budget` | 读 4 / 搜 4 / 测 2，超出要解释 |
| `Medium+` | `Phase Gate` | 以阶段目标和 blocking checks 控制 |
| Bug 根因分析 | `Phase Gate` | 以假设循环、patch 轮次、receipt 控制 |
| harness / infra / CI / 生成器 / 目录迁移 | `Phase Gate` | 以阶段 gate 和兼容检查控制 |

### 超限与升级机制

`Hard Budget`

1. 运行时计数
2. 触达预算时立即停止
3. 说明为什么不够
4. 未获批准不继续

`Soft Budget`

1. 运行时计数
2. 触达阈值时必须报告“为什么任务已超出最初判断”
3. 若下一步会扩大模块边界、切换假设或新增验证集，必须先得到确认
4. 若只是完成当前已锁定路径上的最后一小步，可继续收尾，但不得静默扩散

`Phase Gate`

1. 不以文件数/搜索轮次触发自动停机
2. 以阶段 blocking checks 失败作为停机条件
3. 当前阶段未生成 receipt，不得宣称完成
4. 当前阶段 gate 未通过，不得进入下一阶段

静默超预算 = 违规。

## 任务类型路由

- **新功能**：范围由需求定义，直接定位目标代码实现。不需要在整个项目里"排查"相关逻辑。
- **Bug 修复**：需要先定位根因（遵循 dev-principles 第 3 条）。如果是简单、边界明确的问题，可用 `Soft Budget`；如果进入 RCA、flaky、跨模块或环境相关问题，应切到 `Phase Gate`。
- **重构**：先明确重构边界，列出所有受影响文件后再动手。
- **脚本 / CI / 生成器 / 目录迁移 / harness 优化**：默认使用 `Phase Gate`，不应主要依赖文件计数做停机。

### Bug / RCA 的额外约束

对 Bug 和根因分析任务，真正应被限制的不是“读了几个文件”，而是：

- 假设循环次数
- 无效 patch 轮次
- 是否已经产出 root-cause receipt

推荐控制：

1. 连续 2 轮修复失败，必须切入更显式的根因分析
2. 出现 flaky / 环境相关 / 跨模块信号时，直接升级为 `Phase Gate`
3. 根因未明确前，状态应是 `investigating` 或 `blocked`，不能假装完成

### Harness / Infra / Meta 优化的额外约束

对脚本、生成器、CI、目录迁移、harness 本身的优化：

- 主控制器应是 `Phase Gate`
- 每个 Phase 必须有 blocking checks
- 当前 Phase 的 gate 未通过，不得推进下一 Phase
- 预算仅作为提醒，不应成为主停机条件

---

## Micro 快车道

改动规模为 Micro 且风险区域非 Critical 时，执行以下精简流程：

1. **锁定边界**：最多 2 个文件（1 实现 + 1 测试），不读其他文件。
2. **搜索**：最多 2 轮（`rg` 定位 + 精读命中片段），不做广泛排查。
3. **实现**：先做最小可行改动，不顺手扩展、不加额外功能。
4. **验证**：只跑目标测试，不跑模块全量或项目全量。
5. **异常快速恢复**：遇到 sandbox / testmanager 等环境限制，直接切授权重跑，不走二次排错流程。

目标：Micro + Standard/Low 任务 ≤ 5 分钟完成。

---

## Phase Gate 模式的最低要求

当任务进入 `Phase Gate` 模式时，Scope Card 中至少应额外声明：

- 当前阶段目标
- 当前阶段 blocking checks
- 当前阶段完成标准
- 是否允许推进下一阶段

最低约束：

1. 不能等所有阶段做完再统一验证
2. 每个阶段结束后都必须先跑本阶段 gate
3. gate 不通过，状态只能是 `needs-fix` / `blocked`
4. 没有 phase receipt，不得进入下一阶段

---

## 完成回执

任务完成时，按执行模式输出回执：

```
📊 预算消耗
─────────────────────────────
文件读取：N / 上限 M
搜索轮次：N / 上限 M
测试执行：N / 上限 M
是否超限：否 / 是（已获批准）
─────────────────────────────
```

或在 `Phase Gate` 模式下输出：

```
📋 Phase Receipt
─────────────────────────────
阶段：Phase X
Blocking Checks：通过 / 失败
Required Receipt：已生成 / 未生成
是否可进入下一阶段：是 / 否
Waiver：无 / 有（列出）
─────────────────────────────
```

---

## 中断恢复

会话中断重连后：

1. 重新读取 Scope Card（如果上下文丢失则重新输出）。
2. 仅恢复 Scope Card 范围内的文件上下文。
3. 不重新执行已完成的检索，不从头开始排查。
